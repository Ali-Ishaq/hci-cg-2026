The Problem (Dix et al., HCI 3rd ed., p. 8 – Figure 0.1)

A mechanical syringe was being developed and a prototype was demonstrated to hospital staff. They quickly noticed a potentially fatal flaw in its interface. The task is setting the dose to 1372. The book compares two interfaces: before and after user involvement (Dix et al., Design Focus “Quick fixes”; Figure 0.1).

  • Before consulting users: Doses were entered via a numeric keypad (like a telephone or calculator). One accidental keypress and the dose could be out by a factor of 10. For example, a slip could produce 13672 instead of 1372.
  • After consulting users: The production version had individual increment/decrement buttons for each digit (four columns, each with + and −). A mistake then affects at most one digit by one unit, not the whole number (more about participatory design in Chapter 13; Dix et al.).

The lesson: Think “user”, try it out, involve the users, iterate. Involving actual users (hospital staff) made the fatal flaw visible and led to a safer design.

Analysing the Two Interfaces with Norman, GOMS, and KLM

The purpose of Lab 01 is to clearly communicate the differences between Norman’s model, GOMS, and KLM. Below, the same task (set dose to 1372) is analysed in all three frameworks for both the keypad design and the increment/decrement design.

Norman’s Model: The Seven Stages of the Action Cycle (Dix et al., Ch. 3, pp. 124–127)

Norman’s model describes the user’s journey in seven stages: Goal → Plan → Specify → Execute → Perceive → Interpret → Compare (Dix et al., Ch. 3, pp. 124–127). The table below applies these stages to both syringe interfaces.

Norman’s seven stages applied to the Automatic Syringe (Before vs After user involvement)
Norman’s stage Before (numeric keypad) After (increment/decrement per digit)
1. Goal Set dose to 1372. Set dose to 1372.
2. Plan “I will type the four digits on the keypad.” “I will set each digit column to 1, 3, 7, 2 using +/−.”
3. Specify User must specify which keys (1, 3, 7, 2) and in what order. One wrong or extra key (e.g. 6 instead of 7, or double 2) changes the outcome drastically. User specifies which digit column and whether to increment or decrement. Actions are constrained: only change one digit at a time by one unit.
4. Execute Press four keys. Simple motor action, but a single slip (wrong key or extra key) can produce e.g. 13672. Multiple button presses per digit. More actions, but each action has a small, predictable effect.
5. Perceive User sees the displayed number (e.g. 13672). The display does not highlight which digit might be wrong. User sees each digit change as they press +/−. Feedback is per digit; each change is visible immediately.
6. Interpret User must interpret “what number is this?” and “is it 1372?” A wrong number can look plausible at a glance. User interprets each digit’s value. Easier to interpret because each digit is set and confirmed separately.
7. Compare User compares display (e.g. 13672) with goal (1372). May not notice the slip; comparison is error-prone. User compares each digit to the intended value. Mismatches are easier to spot (e.g. “that digit should be 7, not 6”).

Gulf of Execution and Gulf of Evaluation (Dix et al., Ch. 3, pp. 126–127)

Gulf of Execution is the gap between the user’s goal and the actions the system allows. With the keypad, the gulf is wide: the user can easily form the wrong intention (e.g. press 6 instead of 7) or make a motor slip (extra key), and the system does not constrain or correct—one slip can put the dose out by a factor of 10. With increment/decrement, the gulf is narrower: the interface only allows small steps per digit, so large errors are much harder to make.

Gulf of Evaluation is the gap between the system’s state and what the user can perceive and interpret. With the keypad, the gulf is wide: a wrong number (e.g. 13672) can look plausible; the user may not detect the error before it is too late. With increment/decrement, the gulf is narrower: each digit is adjusted and visible separately, so mismatches between goal and display are easier to detect. Norman’s model thus explains why involving users led to a design that narrows both gulfs.

GOMS: Goals, Operators, Methods, Selection Rules (Dix et al., Ch. 12, pp. 422–430)

GOMS models the task structure for expert users: what is the goal, what operators (primitive actions) are used, what methods (sequences of subgoals and operators) achieve the goal, and what selection rules choose between methods.

GOMS applied to the Automatic Syringe (Before vs After)
GOMS concept Before (numeric keypad) After (increment/decrement per digit)
Goal Set-dose-to-1372 (top-level goal). Set-dose-to-1372 (top-level goal).
Method Keypad-entry: Subgoals = press key “1”, press key “3”, press key “7”, press key “2”. Sequence of four keystroke operators. Increment-decrement: Subgoals = set-digit-1-to-1, set-digit-2-to-3, set-digit-3-to-7, set-digit-4-to-2. Each subgoal is achieved by a sequence of Point + Button (or Keystroke) operators (press + or − repeatedly).
Operators Keystroke (K) or button press (B): one per digit. Four primitives. A single wrong operator (e.g. K[6] instead of K[7]) changes the outcome by a large amount. Point (P), Button (B): many operators (e.g. point to column, press + several times). Each operator changes one digit by one; a wrong operator has bounded effect (one digit off by one).
Selection rules If dose is entered by keypad → use Keypad-entry method. (Only one method modelled here.) If dose is set by increment/decrement → use Increment-decrement method. (Only one method modelled here.)
What GOMS highlights Task is short (4 operators) but error-sensitive: one wrong operator → wrong dose, possibly by factor of 10. GOMS does not model perception or evaluation; it assumes the user executes the method. Task has more operators but error-tolerant: one wrong operator affects only one digit. GOMS again does not model evaluation; the benefit of the redesign is in the structure of the method (smaller effect per operator).

KLM: Keystroke-Level Model and Execution Time (Dix et al., Ch. 12, pp. 436–441)

KLM uses primitive operators (K, B, P, H, M, R, etc.) to predict execution time for expert, error-free performance. It does not model planning or interpretation; it focuses on the physical and mental preparation for each action (Dix et al., Ch. 12).

KLM applied to the Automatic Syringe (Before vs After)
Aspect Before (numeric keypad) After (increment/decrement per digit)
Operator sequence (simplified) [M] [K] [K] [K] [K] — one mental preparation then four keystrokes (for 1, 3, 7, 2). Per digit: [M] [P] [B] (or [K]) repeated. E.g. digit 1: [M][P][B] (0 or 1 press); digit 2: [M][P][B][B][B] (2 presses from 1→3); digit 3: [M][P] then 6×[B]; digit 4: [M][P][B]. Many more M, P, B operators.
Predicted time (conceptually) Short: ~4×K + 1×M (e.g. ~1.35 + 4×0.2 ≈ 2.15 s if K≈0.2 s, M≈1.35 s). Fast but brittle. Longer: many M, P, B. E.g. 4×(M+P+B) plus multiple B per digit. Slower for expert, but one “slip” in KLM (one extra or wrong B) changes result by only one unit in one digit.
What KLM does not model KLM does not model the consequence of a slip: that one wrong K can mean 13672 instead of 1372. It only predicts time for correct execution. Same: KLM does not model detection of error or recovery; it only counts operators for the correct sequence. The design’s benefit (bounded error) is about error consequence, not raw time.
Takeaway Norman explains why the keypad is dangerous (gulfs, feedback, one slip → big error). GOMS shows the method structure (few operators, high error impact). KLM predicts the keypad is faster in error-free execution but does not explain the fatal flaw; that requires Norman’s evaluation side and the real-world consequence of a slip. KLM predicts the increment/decrement design is slower in expert, error-free time. The reason to prefer it is safety (Norman: narrower gulfs; GOMS: each operator has bounded effect), not speed.

Summary: How the Three Models Differ on This Problem

Differences between Norman, GOMS, and KLM in the Automatic Syringe case
Model What it explains or predicts Automatic Syringe (keypad vs +/−)
Norman Understanding, errors, feedback, gulfs of execution and evaluation. Keypad: wide gulfs (one slip → wrong dose; hard to notice). +/−: narrower gulfs (constrained actions; per-digit feedback). Explains why user involvement led to a safer design.
GOMS Task structure: goals, methods, operators, selection rules. Relative efficiency of methods. Keypad: short method (4 operators), but one wrong operator → large error. +/−: longer method, but one wrong operator → small error. Highlights structure and error impact of the method.
KLM Execution time for expert, error-free performance (primitive operators K, B, P, M, etc.). Keypad: fewer operators → faster predicted time. +/−: more operators → slower. KLM does not predict the fatal flaw; it only predicts time. The design choice (involve users, iterate) is justified by Norman and by the real-world consequence of slips, not by KLM alone.
