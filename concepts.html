Overview

This site explores fundamental HCI concepts from Human-Computer Interaction (Dix, Finlay, Abowd, & Beale, 2004, 3rd ed.). We focus on cognitive models that explain how users perceive, interpret, and act within digital systems, and how designers can evaluate and improve interaction.

Cognitive & Task Models

These models help us understand and predict user behavior during interaction. They form the foundation for comparing different interface designs and task strategies.

Norman's Action Cycle

The action cycle (Norman, 1988) is a seven-stage model describing how users achieve goals through interaction:

  • Execution side: Goal → Plan → Specify → Perform
  • Evaluation side: Perceive → Interpret → Compare

Two critical concepts emerge from this model: the Gulf of Execution (distance between user intention and system actions) and the Gulf of Evaluation (distance between system state and user perception/understanding).

GOMS (Goals, Operators, Methods, Selection Rules)

GOMS provides a structured framework for modeling expert task performance. It decomposes a task into:

  • Goals: What the user wants to achieve
  • Operators: Atomic actions (keystroke, mouse click, cognitive step)
  • Methods: Sequences of operators and subgoals that accomplish a goal
  • Selection Rules: Decision logic for choosing among multiple methods

GOMS is especially useful for comparing interface alternatives and predicting relative task efficiency.

KLM (Keystroke-Level Model)

The Keystroke-Level Model is a specialized, quantitative instance of GOMS that predicts execution time for expert, error-free users. It uses six primitive operators:

  • K: Keystroke
  • B: Button press
  • P: Pointing (mouse move)
  • H: Homing (moving hand to keyboard or mouse)
  • D: Drawing (e.g. line segment)
  • M: Mental operator (cognitive preparation)

Each operator has an empirically derived execution time (in seconds). KLM does not model planning, learning, or error recovery.

Design Principles & Heuristics

Schneiderman's Eight Golden Rules of Interface Design

Shneiderman & Plaisant (2010, adapted from Shneiderman, 1989) offer eight design principles to support usability and user satisfaction:

  1. Strive for consistency – Consistent sequences, terminology, and layout reduce cognitive load
  2. Enable frequent users to use shortcuts – Power users need accelerators (hotkeys, abbreviations) for efficiency
  3. Offer informative feedback – Users must perceive and understand system state at all times
  4. Design dialogs to yield closure – Clear beginning, middle, and end; confirm completion
  5. Prevent errors – Make errors unlikely or automatically recoverable
  6. Permit easy reversal of actions – Undo and redo reduce anxiety and encourage exploration
  7. Support locus of control – Users should feel in control; avoid modes that restrict interaction
  8. Reduce short-term memory load – Keep menus visible, label options, minimize recall demands

Case Studies & Applications

Real-world examples demonstrate how these models and principles work in practice and reveal the consequences of good and poor interaction design.

Automatic Syringe: From Calculator to Controller

A medical syringe for delivering precise doses is redesigned based on user feedback. The original interface uses a numeric keypad (error-prone; one slip can multiply dose by 10×). The improved interface uses digit-specific increment/decrement buttons, making errors local and recoverable. This case illustrates participatory design, the gulfs of execution and evaluation, and how interface constraints can prevent catastrophic user error.

The "Who Paid?" Problem: Splitwise Social Coordination

Splitwise is a social app for expense sharing among friends. The "Who Paid?" problem arises when the interface fails to make shared state clear: "Who paid? Who owes? Is it settled?" Poor design widens the Gulf of Evaluation, creating emotional friction and trust erosion. This case shows how interaction design is not just about tasks, but also about social and emotional consequences.

A practical comparison of two smartphone interfaces for the same task. This analysis demonstrates how GOMS decomposition reveals differences in method selection, operator count, and mental workload—and why one design may be more efficient than another, even when both are "correct."

How These Concepts Connect

Cognitive models (Norman's action cycle, GOMS, KLM) let us understand and predict user interaction.

Design principles (Schneiderman's eight rules) provide actionable guidance for making interfaces usable and satisfying.

Case studies show how models and principles apply in real situations—and what goes wrong when they're ignored.

Together, they form a framework for thinking before sketching: analyzing user tasks, reasoning about interface alternatives, and iterating toward designs that serve users well.
